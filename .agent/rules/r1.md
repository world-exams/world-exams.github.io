# GitCore Rule R1

Esta regla adapta `.agent` al protocolo GitCore usado por `worldexams`.
No define autoridad nueva: traduce la capa raíz para herramientas que consumen `.agent`.

## Prime Directive

- El estado de largo plazo vive en GitHub Issues.
- La arquitectura manda sobre instrucciones ambiguas.
- Los cambios deben ser atómicos, trazables y alineados al monorepo.

## Archivos canónicos del protocolo

Leer en este orden antes de trabajo sustancial:

1. `AGENTS.md`
2. `.gitcore/ARCHITECTURE.md`
3. `.gitcore/AGENT_INDEX.md`
4. `.gitcore/features.json`
5. `.gitcore/planning/PLANNING.md`
6. `.gitcore/planning/TASK.md`
7. `docs/README.md`
8. `docs/monorepo/REPO_AUTHORITY_MATRIX.md`
9. `docs/specs/ACTIVE_PROTOCOLS.md`

## Adaptación local obligatoria

En `worldexams`, la carpeta `.gitcore/planning/` ya existe y es la ubicación válida del planning operativo.

- no inventar trackers paralelos en la raíz;
- no usar `PLANNING.md` o `TASK.md` legacy como autoridad por defecto;
- usar GitHub Issues como estado principal externo;
- usar `.gitcore/planning/`, `docs/monorepo/*`, `docs/specs/*` y `.gitcore/*` como capa canónica vigente.

## Startup loop

Antes de implementar cambios relevantes:

1. Verificar arquitectura en `.gitcore/ARCHITECTURE.md`.
2. Verificar routing/contexto en `.gitcore/AGENT_INDEX.md`.
3. Verificar release/features en `.gitcore/features.json`.
4. Verificar si el trabajo ya está representado en un Issue.
5. Cargar solo la documentación del paquete afectado.

## Workflow obligatorio

- Issue-first para trabajo nuevo.
- Una rama por unidad de trabajo.
- Un commit por cambio lógico.
- Referenciar el issue en commit o PR cuando aplique.
- Actualizar la capa canónica, no notas sueltas, cuando cambien decisiones persistentes.

## Commit contract

Formato preferido:

```text
<type>(<scope>): <descripcion> #<issue>
```

Reglas:

- no mezclar fixes, docs y refactors no relacionados en un solo commit;
- no hacer commits gigantes;
- no usar commits sin scope si el cambio pertenece a un paquete o dominio claro.

## Reglas del repo

- Repo privado y prelaunch.
- No asumir despliegue ni sincronización de repos públicos.
- Monorepo multi-país con lógica/UI compartidas.
- Nuevos países se resuelven por configuración, contenido, branding y SEO.
- No crear GitHub Actions para deploy.
- Deploy manual por CLI (`wrangler`, `supabase`, herramientas locales).

## Protocolo de preguntas

- Validar `docs/specs/ACTIVE_PROTOCOLS.md` antes de trabajo de contenido.
- Para trabajo nuevo, el protocolo base es `docs/QUESTION_GENERATION_PROTOCOL_V3.md`.
- Protocolos legacy solo valen como compatibilidad o contexto histórico explícito.

## Forbidden drift

No hacer esto:

- crear `TODO.md`, `TASK.md`, `BACKLOG.md`, `STATUS.md` o equivalentes fuera de `.gitcore/planning/`;
- tratar documentos legacy como autoridad final;
- duplicar lógica por país creando forks innecesarios;
- exponer secretos o usar claves de servidor en cliente;
- desplegar por GitHub Actions.
